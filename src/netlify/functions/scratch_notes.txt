That's a clean SaaS model. Here's how I'd think about structuring it:

Tier 1 — Free / Preview

Temporary Netlify deploy under your account (expires after 48–72h)
Download the ZIP (you already have this)
Watermark or footer: "Generated by [Your Brand]"
Resume analysis + template extraction included
Tier 2 — Permanent (paid)

Deploy to the user's own GitHub Pages via OAuth
Custom subdomain option (jane.yoursite.com) via your domain + Netlify
No watermark
Ability to regenerate / refresh the site (new job target, updated resume)
PDF and headshot properly embedded
Tier 3 — Premium

Custom domain (janesmith.com) pointed at the hosted site
Multiple versions / A-B variants for different job targets
Priority generation queue
Key implementation decisions to make before building:

Auth — do you want users to have accounts at all? Without accounts you can't tie a permanent deployment to a user. Netlify Identity or Auth0 are natural fits since you're already on Netlify.

Regeneration — if a user updates their resume and wants a fresh site, the system needs to remember their previous inputs (colors, template choice, job target). That means storing the form payload, not just the HTML output.

Expiry enforcement — Netlify's API lets you delete a site programmatically. A scheduled function (cron) could clean up expired free-tier deploys.

Watermarking — easiest to inject a <div> into the generated HTML before deploying rather than asking the AI to include it, since the AI is inconsistent about it.

Worth sketching the auth and storage story before touching the deploy code — everything else follows from those two decisions.
